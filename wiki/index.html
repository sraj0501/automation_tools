Git Integration

Automatic Git repository monitoring and intelligent commit detection

How Git Monitoring Works

1

Filesystem Watching

DevTrack uses fsnotify to watch the .git directory for changes in real-time. It specifically monitors the .git/HEAD file and .git/refs directory.

2

Commit Detection

When a commit is detected, DevTrack extracts commit info using go-git library:

  • Commit hash
  • Author name and email
  • Commit message
  • Timestamp
  • Changed files
3

Smart Prompting

A Terminal UI prompt appears with the commit message pre-filled, asking if you want to log work for this commit. You can edit the message or add more details.

4

Automatic Task Matching

The commit message is parsed with NLP to extract task IDs, then matched against Azure DevOps work items using fuzzy and semantic matching.

Git Monitor Architecture

GitMonitor Component

The GitMonitor struct in Go handles all Git operations:

git_monitor.go 
type GitMonitor struct {
    repoPath string
    repo     *git.Repository
    watcher  *fsnotify.Watcher
    stopChan chan bool
}

type CommitInfo struct {
    Hash      string
    Message   string
    Author    string
    Timestamp time.Time
    Files     []string
}

Key Features:

  • Non-blocking async monitoring with goroutines
  • Debouncing to avoid duplicate detection (100ms delay)
  • Filters out lock files and temporary Git files
  • Graceful shutdown with stop channel

Commit Message Parsing

Automatic Extraction

DevTrack uses regex patterns to extract task identifiers from commit messages:

Supported Formats 
# GitHub-style issue numbers
git commit -m "Fixed #123"
git commit -m "Resolves #456, closes #789"

# Jira/Azure-style tickets
git commit -m "PROJ-123: Add authentication"
git commit -m "AB-456 Fix login bug"

# Multiple patterns
git commit -m "PA-789: Refactored API (closes #234)"

# Time estimates in commits
git commit -m "Fixed #123 [2h]"
git commit -m "AB-456: Added tests (45min)"

Extracted Information:

  • Task IDs: #123, PROJ-456, AB-789
  • Action verbs: Fixed, Resolves, Closes, Add, Refactored
  • Time estimates: 2h, 45min, 1.5 hours
  • Status: Completed, In Progress, etc.

Troubleshooting Git Monitoring

🔴 Commits not detected

Symptoms: You commit but no prompt appears

Solutions:

  1. Check daemon is running: devtrack status
  2. Verify you're in monitored repo: pwd
  3. Check logs for errors: devtrack logs
  4. Restart daemon: devtrack restart
  5. Manually trigger: devtrack force-trigger

🔴 Duplicate prompts

Symptoms: Multiple prompts for same commit

Solutions:

  1. Check only one daemon instance running: ps aux | grep devtrack
  2. Stop all instances: devtrack stop
  3. Start fresh: devtrack start

🔴 Slow detection

Symptoms: Delay between commit and prompt

Solutions:

  1. Normal delay is 100ms (debouncing)
  2. On slow systems, may take up to 1-2 seconds
  3. Check container resources: docker stats devtrack-app
  4. Large repos may have slower detection

Supported Patterns:

  • Ticket formats: #123, PROJ-456, AB-789, ticket: 123
  • Time expressions: 2h, 30min, 1.5 hours, 2 days
  • Action verbs: Fixed, Completed, Started, Working on, Reviewed, Merged, Deployed
  • Status indicators: Done, In Progress, Blocked, Waiting

Task Matching Strategies

Exact ID Match

Highest Priority - 100% Confidence

# Input
"Fixed AB-234"

# Matches
Task AB-234: "Fix authentication bug in login"
Confidence: 1.0 (exact ID match)

Direct task ID references always match first.

Fuzzy String Matching

Handles typos and variations using fuzzywuzzy

# Input
"Fixed autentication bug"  # Note: typo

# Available Tasks
AB-234: "Fix authentication bug in login"
AB-235: "Add authorization checks"

# Match
Task AB-234 (fuzzy score: 89%)
Confidence: 0.89

Algorithms used:

  • Token Sort: Handles word order differences
  • Partial Ratio: Matches substrings
  • Levenshtein Distance: Character-level similarity

Semantic Similarity

Understands meaning using sentence-transformers

Example - Understanding Intent 
# Input
"Users can't sign in with Microsoft accounts"

# Available Tasks
AB-234: "Fix SSO authentication with Azure AD"
AB-235: "Add multi-factor authentication"
AB-236: "Update login page styling"

# Semantic Match
Task AB-234 (cosine similarity: 0.87)
Reason: Semantic understanding links "sign in", "Microsoft accounts" 
        with "SSO authentication", "Azure AD"
Confidence: 0.87

Model: all-MiniLM-L6-v2 (384-dimensional embeddings)

Benefits: Matches tasks even when using different words to describe same issue

Troubleshooting

Common issues and solutions

Common Issues

🔴 Container won't start

Symptoms: docker compose ps shows containers as "Exited"

Solutions:

  1. Check Docker is running: docker info
  2. View error logs: docker compose logs
  3. Rebuild: docker compose build --no-cache
  4. Remove and recreate: docker compose down && docker compose up -d

🔴 devtrack command not found

Symptoms: Shell says command not found

Solutions:

  1. Ensure wrapper is in PATH: echo $PATH | grep .local/bin
  2. Reload shell: source ~/.bashrc or source ~/.zshrc
  3. Re-run installer to deploy wrapper

🔴 Ollama health check failing

Symptoms: devtrack-app won't start, waiting for ollama

Solutions:

  1. Check Ollama is responding: docker exec devtrack-ollama ollama list
  2. Restart service: docker compose restart devtrack-ollama
  3. Check port 11434 is not in use on host
  4. Wait 30-60 seconds for Ollama to initialize

🔴 Workspace not accessible

Symptoms: Git monitoring not detecting commits

Solutions:

  1. Verify workspace exists: ls ~/workspace
  2. Check mount in container: docker exec devtrack-app ls /workspace
  3. Ensure WORKSPACE_PATH is set correctly in installer
  4. Restart containers after creating workspace directory

Getting Help

If you're still experiencing issues:

  • Check docker compose logs -f for detailed error messages
  • Verify Docker Desktop has sufficient resources (4GB+ RAM recommended)
  • Review container inspect output: docker inspect devtrack-app
  • Try complete reinstall with ./uninstall.sh then ./install_dependencies.sh

Integrations Setup

Connect DevTrack with Azure DevOps, GitHub, and Microsoft Graph

Azure DevOps Integration

Step 1: Create Personal Access Token

  1. Navigate to Azure DevOps: https://dev.azure.com/{your-organization}
  2. Click User Settings → Personal Access Tokens
  3. Click + New Token
  4. Configure:
    • Name: DevTrack Integration
    • Expiration: 90 days (recommended)
    • Scopes: Work Items (Read & Write), Code (Read)
  5. Copy the token immediately

Step 2: Configure DevTrack

# Set environment variable
export AZURE_DEVOPS_PAT="your-pat-token"

# Configure via CLI
devtrack config set azure.organization "your-org-name"
devtrack config set azure.project "your-project-name"

# Test connection
devtrack test azure

Features Available

  • Fetch assigned work items
  • Match commits to work items
  • Update work item status
  • Add comments to work items
  • Log time spent

GitHub Integration

Step 1: Create Personal Access Token

  1. Go to GitHub Settings: https://github.com/settings/tokens
  2. Click Generate new token (classic)
  3. Select scopes: repo, read:user, read:org
  4. Copy the token

Step 2: Configure DevTrack

# Set environment variable
export GITHUB_TOKEN="ghp_your_token"

# Configure
devtrack config set github.username "your-username"

# Test connection
devtrack test github

Step 1: Register Azure AD Application

  1. Go to Azure Portal: https://portal.azure.com
  2. Navigate to Azure Active Directory → App registrations
  3. Click + New registration
  4. Name: DevTrack, Redirect URI: (leave blank)
  5. Copy Application (client) ID and Directory (tenant) ID

Database Management

Manage DevTrack's SQLite database, backups, and queries

Database Overview

DevTrack uses SQLite for local data storage:

Location: ~/.local/share/devtrack/devtrack.db
Format: SQLite 3.x
Typical Size: 2-10 MB

Database Schema

Main Tables

Table Purpose Typical Rows
commits Git commit history 100-1000+
tasks Work items and tasks 50-500
time_entries Time tracking records 100-1000+
activities Daily activity log 50-500

Common Queries

View Recent Commits

sqlite3 ~/.local/share/devtrack/devtrack.db "
SELECT 
    substr(commit_hash, 1, 8) as hash,
    datetime(timestamp) as time,
    author,
    message
FROM commits
ORDER BY timestamp DESC
LIMIT 10;"

Tasks by Status

sqlite3 ~/.local/share/devtrack/devtrack.db "
SELECT 
    status,
    COUNT(*) as count,
    SUM(actual_hours) as total_hours
FROM tasks
GROUP BY status
ORDER BY count DESC;"

Daily Time Summary

sqlite3 ~/.local/share/devtrack/devtrack.db "
SELECT 
    DATE(start_time) as date,
    SUM(duration_minutes) / 60.0 as hours,
    COUNT(DISTINCT task_id) as tasks_worked
FROM time_entries
WHERE start_time >= DATE('now', '-7 days')
GROUP BY DATE(start_time)
ORDER BY date DESC;"

Backup & Restore

Create Backup

# Simple backup
cp ~/.local/share/devtrack/devtrack.db \
   ~/backups/devtrack-$(date +%Y%m%d).db

# Compressed backup
sqlite3 ~/.local/share/devtrack/devtrack.db ".backup /tmp/devtrack.db"
gzip /tmp/devtrack.db
mv /tmp/devtrack.db.gz ~/backups/

Restore from Backup

# Stop daemon first
devtrack stop

# Restore
cp ~/backups/devtrack-20241114.db \
   ~/.local/share/devtrack/devtrack.db

# Restart
devtrack start

Database Maintenance

Vacuum (Reclaim Space)

sqlite3 ~/.local/share/devtrack/devtrack.db "VACUUM;"

Analyze (Update Statistics)

sqlite3 ~/.local/share/devtrack/devtrack.db "ANALYZE;"

Check Integrity

sqlite3 ~/.local/share/devtrack/devtrack.db "PRAGMA integrity_check;"

Data Export

Export to CSV

# Export tasks
sqlite3 -header -csv ~/.local/share/devtrack/devtrack.db \
  "SELECT * FROM tasks;" > tasks.csv

# Export commits
sqlite3 -header -csv ~/.local/share/devtrack/devtrack.db \
  "SELECT * FROM commits;" > commits.csv
Best Practices:
  • Backup database daily (automated recommended)
  • Run VACUUM monthly to reclaim space
  • Keep at least 30 days of backups
  • Test restore procedure regularly

Scheduler & Automation

Configure time-based triggers and automated workflows

Scheduler Overview

The scheduler manages time-based prompts for work updates:

  • Configurable intervals (default: 120 minutes)
  • Work hours awareness
  • Weekend detection
  • Pause/resume functionality

Basic Configuration

~/.config/devtrack/config.yaml 
scheduler:
  enabled: true
  interval_minutes: 120  # Every 2 hours
  work_hours:
    enabled: true
    start: "09:00"
    end: "18:00"
    timezone: "America/New_York"
  weekends:
    enabled: false  # Don't trigger on weekends

Work Hours Configuration

Standard Work Hours

scheduler:
  work_hours:
    enabled: true
    start: "09:00"
    end: "18:00"
    days:
      - monday
      - tuesday
      - wednesday
      - thursday
      - friday

Flexible Schedule

scheduler:
  work_hours:
    schedule:
      monday:
        start: "09:00"
        end: "17:00"
      wednesday:
        start: "10:00"  # Late start
        end: "18:00"
      friday:
        start: "09:00"
        end: "15:00"  # Early finish

Trigger Intervals

Interval Use Case Prompts/Day
60 min Detailed tracking ~8
90 min Balanced ~5
120 min Standard (default) ~4
180 min Light tracking ~3

Best Practices

✨ Recommendations:

  • Enable work hours to avoid off-hours interruptions
  • Use focus time blocks for deep work periods
  • Disable on weekends unless you work weekends
  • Integrate with calendar to avoid meeting interruptions
