DevTrack

Intelligent work tracking for developers who'd rather write code than write reports

The Problem

Context Switching Kills Memory

It's end-of-day standup. Your manager asks, "What did you work on today?" Your mind goes blank.

You definitely made 15 commits, reviewed some PRs, and fixed that production bug... but the details are gone. Lost in context switches and coffee breaks.

Then comes the weekly report. You open Azure DevOps and try to reconstruct your week. How long did that authentication bug take? Was it 2 hours or 4? What was that cryptic commit message from 2 AM?

The Solution

Capture Context When It's Fresh

DevTrack watches your Git commits and prompts you to log work right when you commit - while the context is still fresh.

It uses AI to parse your natural language updates, automatically matches them to Azure DevOps tasks, and generates detailed daily reports.

ğŸ‹ Container Layer

  • Go daemon (compiled binary as devtrack-cli)
  • Python 3.11 runtime

  • SQLite database

  • Runs llama3.1, llama2, mistral, or phi3 models
  • Isolated from main application
  • Accessible via internal Docker network
  • Persistent model storage in volume

âš™ï¸ Application Layer

Go Daemon (devtrack-cli): Core orchestration engine

  • Git repository monitoring with fsnotify
  • Cron-based scheduling
  • Terminal UI powered by Bubble Tea
  • IPC communication with Python backend
  • SQLite data management

Python Backend: AI and integration layer

  • NLP processing with spaCy
  • Semantic task matching
  • Azure DevOps integration
  • MS Graph API communication
  • Email report generation

ğŸ’¾ Data Persistence

Docker volumes ensure data survives container restarts:

  • devtrack-data: SQLite database, task cache, logs
  • devtrack-config: Configuration files, credentials
  • ollama-data: AI models (can be several GB)
  • workspace: Mounted from host for Git monitoring
âœ¨ Benefits of This Architecture:
  • Platform Independent: Same containers run on Linux, macOS, Windows

  • Resource Efficient: Go provides low overhead, Python handles complex AI tasks
  • Scalable: Can add more containers for additional services

Git Integration

Automatic Git repository monitoring and intelligent commit detection

How Git Monitoring Works

1

Filesystem Watching

DevTrack uses fsnotify to watch the .git directory for changes in real-time. It specifically monitors the .git/HEAD file and .git/refs directory.

2

Commit Detection

When a commit is detected, DevTrack extracts commit info using go-git library:

  • Commit hash
  • Author name and email
  • Commit message
  • Timestamp
  • Changed files
3

Smart Prompting

A Terminal UI prompt appears with the commit message pre-filled, asking if you want to log work for this commit. You can edit the message or add more details.

4

Automatic Task Matching

The commit message is parsed with NLP to extract task IDs, then matched against Azure DevOps work items using fuzzy and semantic matching.

Git Monitor Architecture

ğŸ” GitMonitor Component

The GitMonitor struct in Go handles all Git operations:

git_monitor.go 
type GitMonitor struct {
    repoPath string
    repo     *git.Repository
    watcher  *fsnotify.Watcher
    stopChan chan bool
}

type CommitInfo struct {
    Hash      string
    Message   string
    Author    string
    Timestamp time.Time
    Files     []string
}

Key Features:

  • Non-blocking async monitoring with goroutines
  • Debouncing to avoid duplicate detection (100ms delay)
  • Filters out lock files and temporary Git files
  • Graceful shutdown with stop channel

Troubleshooting Git Monitoring

ğŸ”´ Commits not detected

Symptoms: You commit but no prompt appears

Solutions:

  1. Check daemon is running: devtrack status
  2. Verify you're in monitored repo: pwd
  3. Check logs for errors: devtrack logs
  4. Restart daemon: devtrack restart
  5. Manually trigger: devtrack force-trigger

ğŸ”´ Duplicate prompts

Symptoms: Multiple prompts for same commit

Solutions:

  1. Check only one daemon instance running: ps aux | grep devtrack
  2. Stop all instances: devtrack stop
  3. Start fresh: devtrack start

ğŸ”´ Slow detection

Symptoms: Delay between commit and prompt

Solutions:

  1. Normal delay is 100ms (debouncing)
  2. On slow systems, may take up to 1-2 seconds
  3. Check container resources: docker stats devtrack-app
  4. Large repos may have slower detection

Troubleshooting

Common issues and solutions

Common Issues

ğŸ”´ Container won't start

Symptoms: docker compose ps shows containers as "Exited"

Solutions:

  1. Check Docker is running: docker info
  2. View error logs: docker compose logs
  3. Rebuild: docker compose build --no-cache
  4. Remove and recreate: docker compose down && docker compose up -d

ğŸ”´ devtrack command not found

Symptoms: Shell says command not found

Solutions:

  1. Ensure wrapper is in PATH: echo $PATH | grep .local/bin
  2. Reload shell: source ~/.bashrc or source ~/.zshrc
  3. Re-run installer to deploy wrapper

ğŸ”´ Ollama health check failing

Symptoms: devtrack-app won't start, waiting for ollama

Solutions:

  1. Check Ollama is responding: docker exec devtrack-ollama ollama list
  2. Restart service: docker compose restart devtrack-ollama
  3. Check port 11434 is not in use on host
  4. Wait 30-60 seconds for Ollama to initialize

ğŸ”´ Workspace not accessible

Symptoms: Git monitoring not detecting commits

Solutions:

  1. Verify workspace exists: ls ~/workspace
  2. Check mount in container: docker exec devtrack-app ls /workspace
  3. Ensure WORKSPACE_PATH is set correctly in installer
  4. Restart containers after creating workspace directory

Getting Help

If you're still experiencing issues:

  • Check docker compose logs -f for detailed error messages
  • Verify Docker Desktop has sufficient resources (4GB+ RAM recommended)
  • Review container inspect output: docker inspect devtrack-app
  • Try complete reinstall with ./uninstall.sh then ./install_dependencies.sh
